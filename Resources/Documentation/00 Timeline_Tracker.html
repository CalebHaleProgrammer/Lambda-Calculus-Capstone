My thoughts on Abstraction: abstraction is the process of creating an abstract object, also called an abstraction, which includes some data/definition and has some data/definition which is not defined or included. That which is defined is that which remains constant and is more often the focus in mathematics, because it offers reliable unchanging information that you can use in all cases. That which is not defined is that which can change in the application of an abstraction and is more often the focus in computer science, because if offers flexibility and descriptive power across varied cases.

Cont.: In everyday language groups and classifications are examples of abstractions, such as “dog”, which is defined by some criteria (four legs, tame-able, “man’s best friend”, etc.) and which can vary by undefined criteria (color, who owns it as a pet, breed, etc.). Proving that dogs have more legs than flamingos would use the defined criteria, more akin to a math proof, while a business creating a service that creates custom dog clothes that match their fur color would employ techniques that can vary by the undefined criteria. Thus, both aspects are useful; one for broad, powerful principle-based reasoning, the other for varied, dynamic/flexible situation-based action. One uses a single “variable” (constant) to represent many unchanging values, while the other uses a single variable to represent many changing values.

Cont.: I think this is a strong conceptual formulation, but I’m not completely sure how it aligns with abstraction in the technical sense. Functions in both normal programming and LC are called abstractions, (the “defined” aspect being the code in the function and the “undefined” aspect being the arguments), and the term “abstraction” is also used in programming when talking about a system’s structure: having separate specialized methods that can act as a black box and be used without worrying much about the details of how it is implemented. These allow higher-level reasoning based on functionality (e.g. “determine eligibility”->“report eligibility to user”) similar to how grouping and classification is used in everyday reasoning (e.g. you don’t need to know a dog’s color to determine how many legs you would expect it to have, any more than you’d need to know the workings of a function to know what it does). Thus, referring to functions as abstractions and referring to program abstraction (as a methodology or structure) can use the same idea of abstraction, as I understand it, but take advantage of different aspects (what varies, for functions, what remains the same, for fulfilling functionality needs).

In the second example, each side is its own expression, they are equal not by intension but by extension – they refer to the same thing – which can be seen more clearly by different notation. f1(x)=f2(x) is only equal when the effect of f1 is the same as the effect of f2, that is, when ()^2 = ()+2.

Using the “intensional” definition as something like “what a function does” and “extensional” as “what effect or end result it has” reveals a distinction between two types of intensional definitions, which I think correspond somewhat with simple and dependent types. Thus, a “simply typed” intensional definition corresponds to a single extension, while a “dependently typed” intensional definition corresponds to a whole class or set of extensions. New formulation: Objects are either a single value (constant), a single unknown value (“existential variable” or “false variable”), or a variety of values (“universal variable” or simply “variable”) from a collection that defines it, i.e. a set. What I termed existential variables would likely be called “false variables” but I refer to them as variables since “what they can be” can vary, although in practice they function as an unknown constant. Any of these value(s) can then be stated to belong to another set of values, which further describes the value(s), and which corresponds to a type. Thus, an individual dog can be of type dog, mammal, vertebrate, pet, etc.

Then, abstraction is the process of specifying a constant or variable to be in a larger set than previously defined; C|Ve|Vu + _|type -> Vu + type (such that the type is a larger superset of the previous type, which if not specified would just be the set of the thing itself, it belonging to a collection of nothing but itself at the very least). Of note is that the primary defining term, i.e. the constant or variable itself, becomes a “universal”/true variable because the abstract object is a class that includes the original object. Without this change, the term would simply refer to the same object in more vague/broad terms.

As sets can be defined by set-builder notation or enumerated elements, this corresponds to intensional and extensional definitions, and either can be used in the definition of the primary C|V term and the type term. When the type term’s definition is intensionally based on the primary C|V term, it is a dependent type. When the primary C|V term is a variable defined from another term/obj, then that is a bound variable. Existence is the property of having an element in the extension of a set as defined intensionally (e.g. set builder notation).

This formulation encapsulates well my brainstorming on different levels of detail in description, natures of definition, ways variables (or what we so call) are used, types, and constant descriptions/definitions of variable objects.

:40 More YT research into type theory (mainly computerphile). J Rule says proofs for reflexivity are sufficient to prove equality. Equality is a groupoid, as a group describes the patterns that govern equality for any closed system (closure describable by operation results belonging to the type of the operated elements). J Rule allows any equality to be treated as a groupoid? Using group behavior we can see the impossibility of proving any two equalities are equal because you can’t prove any two integers are equal (integers being elements of a group under addition). Correspondence allows defining any mappable-comparable things as being equal, e.g. any collection of equal size? Natural numbers and integers are the same under this view. However, operations and other structures/patterns can break these correspondences. UIP (Uniqueness of Identity Proof) fails when there are multiple potential correspondences (mappings) – there must be only one correspondence. Since type theory is implementation-independent, it aligns with black-box methodology for function design in programming (plug-and-play, focusing on results at each step and not the process). HoTT idea came about with types as geometric objects and elements of types as the points on the surface, equality as a path from one point to another. Triviality of the uniqueness of equality asks “up to isomorphism” so topologically “same” paths are considered the same and must be non-trivially different to be considered different equalities. The univalence principle then comes into effect for comparing types, as equal if there is a one-to-one correspondence. HoTT expands from set theory with flexibility comparable to how non-euclidean geometry generalizes from the assumptions of euclidean geometry. In a plane there is only one path between two points, but in other geometries there can be more, which can’t be expressed as plain sets.
